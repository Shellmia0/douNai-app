## 一、需求背景：一个常见但未被好好解决的问题

### 1.1 问题是什么

一位在读硕士研究生，论文截止日期临近（初稿仅剩5天），却持续陷入拖延循环：

- 知道该做什么，但**启动不了**
- 不是不想做，而是一想到就**焦虑逃避**
- 曾尝试多种工具（番茄钟、to-do list、多邻国式打卡），均**无法持续**
- 手机刷到麻木，游戏玩到无聊，连"奖励自己"都失去了吸引力

这不是个例。学术拖延（Academic Procrastination）影响着约70-95%的大学生（Steel, 2007），其核心并非"懒惰"，而是一种**情绪调节失败**——用短期回避来缓解面对任务时产生的负面情绪。

### 1.2 为什么现有工具不够

| 工具类型 | 失败原因 |
|---------|---------|
| To-do list | 只记录不驱动，完成感短暂 |
| 番茄钟 | 解决注意力问题，不解决启动问题 |
| 多邻国式打卡 | 冷冰冰的数据提醒，引发厌恶而非动力 |
| 自律App | 依赖意志力，而意志力恰恰是拖延者缺乏的 |

**核心矛盾**：这些工具假设用户"有动力但缺方法"，而拖延者的真实状态是"缺动力且方法无效"。

### 1.3 我们要解决什么

> 设计一个不依赖意志力、能长期使用的个人行为干预工具，帮助用户从"知道该做但做不到"转变为"自然而然地做"。

## 二、用户调研：不问"你想要什么"，问"你是什么样的人"

### 2.1 调研方法论

我们没有做问卷星式的标准量表，而是采用了**半结构化深度访谈**，通过 Telegram 群对话分三轮完成 14 个问题。

设计原则：
- **先理解模式，再设计工具** —— 不急于问"你想要什么功能"，而是先挖掘拖延背后的心理动力学
- **行为考古** —— 不问"你觉得什么有用"，而是问"过去什么情况下你真的行动了"
- **矛盾挖掘** —— 寻找用户口中的矛盾点，那往往是最真实的需求

### 2.2 三轮调研过程

#### 第一轮：拖延模式画像 (Q1-Q6)

目标：理解拖延的**触发机制**和**逃避模式**

| 问题方向 | 挖掘内容 |
|---------|---------|
| 拖延触发 | 什么场景最容易拖延？是特定任务还是全面的？ |
| 情绪状态 | 拖延时的主导情绪是什么？焦虑？空虚？麻木？ |
| 逃避行为 | 拖延时通常做什么？刷手机？游戏？还是发呆？ |
| 启动条件 | 什么情况下能暂时突破拖延？（关键线索！） |
| 过往干预 | 试过哪些方法？为什么失败了？ |
| 自我认知 | 觉得自己拖延的根本原因是什么？ |

**关键发现**：用户提到"组会前能临时行动"——这说明**外部社交压力**是目前唯一有效的启动机制。

#### 第二轮：驱动力结构 (Q7-Q11)

目标：找到可以**设计**的驱动力，而非依赖偶发的外部压力

**Q7 — 社交驱动力验证**

> 问："如果有人每天温和地盯你一下，你会觉得是帮助还是压力？"  
> 答："帮助"

**理论支撑**：这印证了自我决定理论（SDT, Deci & Ryan, 2000）中"关系需求"（Relatedness）对内在动机的驱动作用。用户不抗拒监督，需要的是**"有人在看"的回响感**——一种非评判性的社会存在。

**Q8 — 奖励敏感度**

> 问："什么样的奖励对你有吸引力？"  
> 答："报复性刷手机和玩游戏，想到这些东西反而没有兴奋感...内心特别渴望出去游玩看看自然风光，但总是缺少出去的理由。办的南京公园年卡都快过期了。"

**理论支撑**：这是典型的**享乐适应**（Hedonic Adaptation）。高频低成本的即时奖励（手机、游戏）因过度暴露而边际效用归零。而用户真正渴望的是**体验型奖励**（Experiential Reward）——Van Boven & Gilovich (2003) 的研究表明，体验型消费比物质消费带来更持久的幸福感。

> 🔑 **关键洞察**：用户不需要新的虚拟奖励，需要一个"正当理由"去做她本来就想做的事。

**Q9-Q10 — 监督风格偏好**

> "温柔+严厉地循循善诱或许对我有用"
> "多邻国式冷冰冰的数据提醒没用"

**理论支撑**：这指向了 Vygotsky 的"最近发展区"概念——用户需要的不是放任也不是压迫，而是一个**脚手架**（Scaffolding），在她能力边界处提供恰到好处的支持。

**Q11 — 技术偏好**

> "微信、小红书、抖音（已卸载以避免成瘾）"
> "工具如果能嵌到习惯中当然很好，但也希望不要过度侵犯导致厌恶"

**设计启示**：工具需要**存在但不入侵**——类似环境心理学中"可供性"（Affordance）的概念：提供行动的可能性，但不强制行动。

#### 第三轮：项目约束 (Q12-Q14)

| 维度 | 数据 | 设计含义 |
|------|------|---------|
| 初稿截止 | 3月9日（5天后） | 需要立即可用，不能等"完美版" |
| 完成度 | ~50%，Ch3/4/5需打磨 | 有基础，不是从零开始，信心可建立 |
| 每日时间 | 4-6小时 | 任务拆解需匹配这个容量 |
| 最终提交 | 6月 | 工具需要长期可用，不只是5天 |

### 2.3 用户画像提炼

综合三轮调研，提炼出**6条核心设计指导原则**：

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│   ① 社交驱动有效 ──→ 设计"有人在看"的感觉                     │
│                                                            │
│   ② 虚拟奖励已燃尽 ──→ 连接真实世界体验                       │
│                                                            │
│   ③ 渴望出去但"缺理由" ──→ 探索奖励 = 核心差异化               │
│                                                            │
│   ④ 小任务能启动 ──→ 拆解为基础设施                           │
│                                                            │
│   ⑤ 需要温度不要数据 ──→ 教练型角色，非仪表盘                  │
│                                                            │

```

## 三、设计：从原理到方案

### 3.1 理论框架

DouNai 的设计融合了三个行为科学理论：

#### 自我决定理论 (SDT)

Deci & Ryan 提出人有三种基本心理需求：自主性、胜任感、关系需求。我们的设计回应了每一个：

| 需求 | 设计映射 |
|------|---------|
| **自主性** (Autonomy) | 用户自己选择任务、自己决定何时结算、奖励菜单可自定义 |
| **胜任感** (Competence) | 小任务易完成、日记评估实时反馈进步、连续天数可视化 |
| **关系需求** (Relatedness) | AI陪伴check-in、暖心反馈语、"有人在看"的设计 |

#### 行为设计 (Fogg Behavior Model)

BJ Fogg 的公式：**B = MAP**（行为 = 动机 × 能力 × 提示）

| 变量 | 传统工具的问题 | DouNai 的解法 |
|------|-------------|-------------|
| **动机 (M)** | 依赖意志力（不可靠） | 真实世界探索奖励（内在渴望驱动） |
| **能力 (A)** | 任务太大导致瘫痪 | 拆解为2-3个小任务 |
| **提示 (P)** | 冷提醒引发厌恶 | 有温度的每日3触点 |

#### 操作性条件反射 (Variable Ratio Reinforcement)

固定奖励容易产生适应（"又是这个"），而可变比率强化保持新鲜感：

- 基础🌸是固定的（完成=1🌸），提供**确定性**
- 连续天数bonus是**渐进的**（3天+1, 7天+3）
- 日记评估的🌸是**可变的**（取决于写作深度，0-2.5🌸）
- 奖励菜单有多层多项，每次兑换都是**选择**

### 3.2 核心循环设计

```
    ╭──────────────────────────────────────╮
    │          DouNai 核心行为循环           │
    ╰──────────────────────────────────────╯

    ┌─────────┐         ┌─────────┐
    │  触发    │────────▶│  行动    │
    │ (提示)   │         │ (任务)   │
    └─────────┘         └────┬────┘
         ▲                    │
         │                    ▼
    ┌─────────┐         ┌─────────┐
    │  投资    │◀────────│  奖励    │
    │ (日记)   │         │ (🌸)    │
    └─────────┘         └─────────┘

    参考：Nir Eyal《Hooked》模型
    触发→行动→奖励→投资→下一次触发
```

**触发**：每日定时check-in（"今天打算先碰哪一块？"）  
**行动**：完成小任务（低门槛，高可行性）  
**奖励**：获得🌸，离真实探索更近一步（可变、有意义）  
**投资**：写日记反思（增加沉没成本，提高下次触发效率）

### 3.3 功能模块设计

#### 模块一：任务管理 📋

**设计原则**：让"开始"变得极简

- 添加任务：一行输入框，回车即添加
- 勾选任务：点击圆圈，绿色勾表示完成
- 结算按钮：完成≥50%时出现，不强制100%

![今日页](images/01-today.png)

**为什么不要求100%完成？**  
因为对拖延者来说，"必须全部完成"本身就会成为压力源。50%阈值传递的信息是："做一半也算数，完美不是标准，行动才是。"

#### 模块二：风格化日记 📔

**设计原则**：用仪式感降低启动阻力

用户反馈"感觉体验不直观"后，我们将普通文本框改造为**日记本样式**：

- 大字日期 "3.4" + 星期 + 时间
- 天气选择器（☀️⛅🌧️❄️🌫️）
- 心情选择器（😊😐😫💪😴）+ 选择反馈动画
- 横线信纸背景 + 左侧书脊装订线

![日记本](images/01-today.png)

**为什么需要仪式感？**  
行为触发理论告诉我们，**环境线索**（contextual cue）能降低行为启动成本。日记本的视觉风格本身就是一个触发："现在是写日记的时间了。"

#### 模块三：深度评估引擎 📊

**设计原则**：评估标准透明化，让深度思考"自然发生"

用户提出"鼓励深度思考和学着记录"的需求后，我们设计了5维度评估系统：

| 维度 | 检测方法 | 心理学依据 |
|------|---------|----------|
| ✍️ 字数 | 100字+/300字+ | 写作量与反思深度正相关（Pennebaker, 1997） |
| 🔍 反思 | 关键词检测（因为/发现/意识到） | 因果归因是元认知的核心标志 |
| 🎯 行动 | 关键词检测（下次/打算/计划） | 实施意图（Implementation Intention）提高行动概率 |
| ❓ 自问 | 包含问号 | 自我提问触发深层加工（Deep Processing） |

**为什么公开评分规则？**  
传统做法是隐藏算法让用户"自然表现"，但对于我们的场景：
- 公开规则 → 用户为了高分主动写反思/计划/提问 → 实际上就是在做深度思考
- **"应试"心态在这里是feature，不是bug** — 因为"假装深度思考"和"真的深度思考"在神经层面几乎没有区别

![日记评估](images/05-celebration.png)

#### 模块四：奖励商店 🎁

**设计原则**：奖励必须连接真实世界体验

5档设计不是随意分级，而是遵循了**奖励梯度理论**：

| 档位 | 获取周期 | 心理功能 |
|------|---------|---------|
| 🌸×1 小确幸 | 每天 | **即时满足**，维持日常动力 |
| 🌸×3 小奖励 | 2-3天 | **短期目标**，培养连续感 |
| 🌸×7 中奖励 | ~1周 | **里程碑感**，建立自我效能 |
| 🌸×15 大奖励 | ~2周 | **延迟满足**，训练目标持久力 |
| 🌸×30 终极奖励 | ~1月 | **长期愿景**，连接人生意义 |

![奖励商店](images/02-rewards.png)

**关键规则：攒够了必须兑换**  
这不是可选的——奖励自己也是任务的一部分。因为拖延者的另一个常见模式是"觉得自己不配享受"，而强制兑换打破了这个心理防线。

#### 模块五：数据可视化 📈

**设计原则**：让进步可见，让坚持有重量

- 四宫格总览：总天数、完成任务、最长连续、兑换奖励
- **花开热力图**：类GitHub贡献图，用颜色深浅展示每天的产出密度
- 每日记录流水：含心情、天气、完成度、日记摘要

![记录页](images/03-history.png)

**为什么叫"花开记录"而不是"贡献图"？**  
因为"贡献"暗示义务和工作，而"花开"传递的是**自然生长**——你不是在完成任务，你是在浇灌自己的花园。

## 四、开发迭代：用户说了算

### 4.1 迭代方法论

我们采用**极速用户驱动迭代**，流程如下：

```
用户在 Telegram 群提需求/报 bug
        │
        ▼
Yui 立即修改代码（5-15分钟）
        │
        ▼
git push → 线上自动更新
        │
        ▼
用户刷新页面验证
        │
        ▼
新一轮反馈
```

**单次迭代周期：5-15分钟**。没有PRD、没有评审会、没有排期——用户说完，下一条消息就是更新通知。

### 4.2 完整迭代记录

#### 迭代 #1：MVP 上线
- **时间**: 14:51
- **内容**: 基础版发布（任务管理 + 奖励商店 + 记录页 + 设置页）
- **技术**: HTML + CSS + Vanilla JS，localStorage存储，GitHub Pages部署
- **截图**:

![MVP](images/01-today.png)

#### 迭代 #2：Bug 修复 — 任务只能添加一个
- **时间**: 15:02
- **用户反馈**: 发截图，"只能新增一个任务，第二个无法添加"
- **根因分析**: `emptyTasks` DOM元素在第一次 `innerHTML = ''` 时被销毁，第二次调用 `document.getElementById('emptyTasks')` 返回 null，导致 JS 报错中断
- **修复**: 改为动态生成空状态HTML，不依赖静态DOM元素
- **迭代时间**: 8分钟

#### 迭代 #3：心情选择缺少反馈
- **时间**: 15:13
- **用户反馈**: 发截图，"点击后没什么反馈"
- **分析**: 选择心情后只是border变色，感知不到"发生了什么"
- **修复**: 增加三重反馈——弹跳动画 + 粉色光晕 + 暖心toast提示
- **设计细节**: 每个心情有专属文案（😴→"累了就休息一下～"）
- **迭代时间**: 6分钟

#### 迭代 #4：日记本风格化
- **时间**: 15:20
- **用户反馈**: "希望记录模块是风格化的日记本样式，显示类似'3.14 几点几分 天气：晴朗 心情：可选项'"
- **改造内容**:
  - 大字日期显示（Georgia衬线体）
  - 天气选择器（5种天气）
  - 横线信纸背景（CSS渐变模拟）
  - 左侧书脊装订线（棕色渐变色条）
  - 泛黄纸张色调（#FFFDF7）

#### 迭代 #5：日记深度评估引擎
- **时间**: 14:58（在迭代#4之前提出，并行开发）
- **用户需求**: "增加日记或感想记录模块，根据字数和思考深度发放奖励"
- **实现**: 5星评估系统，含实时预览、提交评估报告、修改退款重评
- **关键设计**: 评分规则完全透明，引导用户主动深度思考

#### 迭代 #6：云端备份
- **时间**: 15:54
- **用户担忧**: "记录存在哪里呢，不会丢失吧"
- **方案设计**: Python HTTP Server + frp公网穿透
- **实现**:
  - Mac Mini本地运行备份API（port 23018）
  - frp隧道暴露到公网（49.232.32.77:23018）
  - 前端自动同步（3秒防抖）
  - 保留最近30个版本
  - macOS launchd自启动
- **迭代时间**: 13分钟

```
│  frps :7256 (控制面板)                             │
│  → :23018 (douNai-app 代理)                       │
└──────────────────────────────────────────────────┘
```

### 5.2 为什么选择零依赖

| 决策 | 理由 |
|------|------|
| 不用 React/Vue | 功能简单，框架增加复杂度而无收益 |
| 不用数据库 | 单用户+小数据量，JSON文件足够 |
| 不用后端框架 | Python标准库http.server能满足需求 |
| 不用第三方CSS | 自定义设计需要完全控制样式 |

**结果**: 总传输大小 < 50KB，首屏加载 < 1秒，零构建步骤。
